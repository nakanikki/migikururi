# クリスタの遅延について

CLIP STUDIO PAINT（クリスタ）では、「右くるり」のようなソフトから送った
ショートカットキーの反映が、物理キーボードで打つより数フレーム遅れます。
このページは、その**原因の調査記録**と、右くるりに実装した**解決策**の説明です。

（計測はすべて 60fps 録画で「キー監視ツールにキーが表示された瞬間 →
クリスタの画面が変化するまで」のフレーム差。1F ≈ 16.7ms）

---

## 現象

| 入力源 | クリスタ反映 |
|---|---|
| 物理キーボード（ロジクール等） | **1F** |
| ソフトからの送出（SendInput） | **4〜6F** |

同じ Ctrl+Z なのに、ソフトから送ると明らかに遅い。
アンドゥ連打などで「ワンテンポ遅れる」と体感できるレベルの差です。

---

## 犯人探しの記録（先に結論: 全部ハズレだった説）

**説1: injected フラグのせい？**
ソフト送出のキーには「注入された入力」のフラグが付くので、クリスタが
差別しているのでは → **ハズレ**。フラグの付かない本物の USB キーボード
（マイコン）から送っても遅かった。

**説2: 送り方（VK / スキャンコード）のせい？**
スキャンコード方式に変えると約1F縮んだ（6F→5F）が、**1F には遠い**。

**説3: マイコン（チップ）の性能のせい？**
Pro Micro（ATmega32U4）を本物の USB キーボードとして使う実験
→ チップが自力で打っても **3F**。ファームを QMK に変えても同じ 3F。
「安いマイコンは 3F が限界」に見えた…が、これも本質ではなかった。
（詳細: [クリスタ無遅延機能について.md](クリスタ無遅延機能について.md)）

---

## 真犯人: 「修飾キーと本キーの同時到着」

決定的だったのは、**Ctrl と Z の間隔を変えながら測った**こと:

| Ctrl→Z の間隔 | クリスタ反映（Z 到着から） |
|---|---|
| ほぼ同時（ソフト送出の通常） | 3〜5F |
| 17ms | 5F（**最悪**） |
| 150ms | **1F** |

**クリスタは Ctrl が押されると内部処理（一時ツール切替など）を始め、
その最中に届いた本キーは処理が終わるまで待たされる**。
Ctrl を十分先に（100ms 以上前に）押しておけば、本キーは 1F で反映される。

つまり——

> **物理キーボードが速いのは、ハードが優秀だからではなく、
> 人間が Ctrl → Z を打つとき自然に 100ms 以上の間隔が空くから。**

ソフトやマイコンは Ctrl と Z を一瞬で連続送出（バースト）するため、
この「クリスタの都合」に毎回ぶつかっていた、というのが真相です。
SendInput でも、修飾キーを 150ms 先行させれば **1F** が出ます（実測済み）。

---

## 右くるりでの解決策

「離すまでキーを送らない」というソフトの設計は守りつつ、**配線で明示的に
組んだ場合だけ**修飾キーを先行できる仕組みを用意しました:

- **特殊キー**（アクション種別）: 修飾キー（Shift/Ctrl/Alt/Space）や
  クリックを送るブロック。「押す（保持）」にした修飾キーは、
  「離す」アクションを置くまで（無ければジェスチャ終了まで）押しっぱなしになる。
- **初期アクションパネル**（「最初に実行」）: ここに繋いだアクションは
  **パイを開いた瞬間**に実行される。

### 設定例（クリスタ用プロファイル）

```
最初に実行 ──◯── 特殊キー: Control（押す）   ← パイを開いた瞬間 Ctrl 先行
アンドゥ ─────── キー: Ctrl+Z               ← 離すと本キーだけ送出 = 1F
リドゥとか ───── 特殊キー: Control（離す）
                └ メニュー: リドゥとか        ← サブメニューに入る時 Ctrl 解除
```

パイを開いて方向を決めるまでの時間（自然に 100ms 以上）が
そのまま Ctrl の先行時間になるので、**離した瞬間 1F で反映**されます。
実測で物理キーボードと同等になりました。

### 限界

- 開いた瞬間に離す超高速フリックでは先行時間が取れず、従来並み（5〜6F）。
  **どんな場合でも従来より遅くなることはありません。**
- 「即時アクション」（ホバー＝発火）のプロファイルでは、発火時点より前に
  先行できないため効果はありません。

---

## まとめ

- クリスタの遅延の正体は「修飾キーと本キーの同時到着」。入力経路や
  ハードの性能ではない。
- 対策は「修飾キーを先に押しておく」だけ。右くるりでは特殊キー＋
  初期アクションの配線でユーザーが自由に組める。
- 仕組みはすべてこのリポジトリのソースで確認できます。
